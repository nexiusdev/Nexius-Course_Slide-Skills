# Nexius Course Template Reference

Use this reference when building Nexius Labs, HARPS, AI transformation, or
agentic-operations course decks. The canonical template is:

- `assets/sample-decks/Day_2_Company_Second_Brain_Template.pptx`
- `assets/visual-reference/day2-company-second-brain-contact-sheet.png`

## Template Identity

The template is an instructor-led course deck, not a marketing presentation. It
uses a dark navy learning environment, structured teaching visuals, frequent
activity/workbook slides, and a repeated bottom message bar that makes the
trainer's takeaway explicit.

Default slide language:

- top-left uppercase section tag in teal,
- top-right official Nexius Labs logo image on every normal content slide,
- large bold white concept title,
- structured body using cards, process tracks, matrices, architecture stacks,
  or a single explanatory illustration,
- bottom bordered takeaway bar labeled with a teaching cue such as `MESSAGE`,
  `REMEMBER`, `TAKEAWAY`, `BOTTOM LINE`, `WORKBOOK`, or `NEXT STEP`,
- tiny muted course footer bottom-left,
- two-digit slide number bottom-right.

## Visual Grammar

Use the template's card-and-diagram grammar as the default:

- Dark navy canvas with slightly lighter navy content panels.
- Teal is the primary structural color for section tags, outlines, progress
  lines, and main message bars.
- Blue signals source, access, query, or information flow.
- Green signals approval, completion, safe use, or permitted action.
- Purple signals review, sensitivity, metadata, or role/context.
- Amber signals ownership, escalation, decision, caution, or unresolved risk.
- Red signals restricted, unsafe, failure, refusal, or a critical gap.

Do not make slides feel like generic SaaS landing pages. The deck should feel
like a high-end classroom operating system: clear, structured, and teachable.

## Brand And Visual Asset Requirements

For Nexius/HARPS course decks, the official logo, icons, and illustrations are
acceptance criteria, not optional polish:

- Use the official Nexius Labs logo image extracted from or supplied with the
  final template. Do not recreate the logo as plain text.
- Put the official logo in the top-right position on every normal content
  slide unless the source template slide intentionally omits it.
- Use icons in cards, flows, matrices, checks, and governance concepts when
  they help the learner scan categories or status. Avoid text-only card grids
  across a whole deck.
- Use the Lucide SVG icon library as the default source for all deck icons.
  Keep stroke weight, color treatment, and container style consistent. Do not
  mix unrelated icon packs unless the user supplies an approved alternative.
- Include actual illustrations on concept-heavy or system-heavy slides. A deck
  that only uses boxes, lines, and text has not followed this template.
- Keep illustration text editable in PowerPoint. The raster image should carry
  the scene or concept, while slide labels and teaching copy remain editable.

## Core Slide Patterns

Use these patterns frequently:

- Title promise: large title, short framing copy, large right-side concept
  illustration, 3-4 compact category pills, and a bottom day promise.
- Output/artifact slide: 4-6 cards showing what participants will create.
- Knowledge debt or problem diagnosis: issue cards with short observable
  symptoms.
- Definition slide: compact equation or flow that turns a term into operational
  meaning.
- Misconception slide: "what it is not" cards with red/amber accents.
- Operating model slide: process path showing inputs, review, answer, evidence,
  and improvement.
- Architecture slide: layered stack or building blocks with one clear role per
  layer.
- Workbook instruction slide: show the artifact fields learners must complete.
- Activity slide: clear task, timebox, output, and debrief cue.
- Quality check slide: criteria cards, test questions, or failure signals.
- Governance slide: permissions, escalation, citation, confidence, audit, or
  feedback loop.
- Rule or management takeaway: one centered statement in a large bordered box
  plus a bottom remember bar.
- Next-step slide: connect today's artifact to the next course day.

## Text And Spacing Rules

Cards should feel seated and calm, not packed.

- Prefer short labels and one concise sentence per card.
- Keep internal padding generous enough that text does not touch borders,
  icons, or adjacent elements.
- If a box looks pinched, shorten text or widen the box before shrinking font.
- Use trainer notes for detail that does not belong on the visible slide.
- Avoid long bullet lists. Convert them into cards, process steps, matrices,
  or a workbook/activity instruction.

The bottom message bar should carry the instructor's punchline. Do not leave it
as decoration; write it as the sentence the trainer wants learners to remember.

## Illustration Use

The template uses illustrations selectively, but a course deck should still
include real explanatory illustrations where they improve understanding. Do not
force a bitmap image on every slide, and do not deliver a substantial concept
deck with no illustrations.

Use an illustration when it helps explain a system, transformation, governance
path, or architecture faster than text. Use editable diagrams when the lesson is
primarily about sequence, roles, criteria, or decisions. Keep teaching labels
editable in PowerPoint whenever possible.

For each visual-heavy slide, compare three possible routes before selecting:

1. a generated dark-course illustration,
2. an editable/native PowerPoint diagram or adapted in-project reference,
3. a licensed third-party/vector marketplace candidate.

Choose the route with the strongest relevance and explanatory power, not the
prettiest image. If a marketplace asset has useful structure but uncertain
license status, redraw or regenerate an original template-style version.

## Trainer Notes

Every instructional slide must include speaker notes. Notes should say how to
teach the slide, what question to ask, what activity instruction to give, what
debrief point to listen for, or how to transition to the next slide. Trainer
notes are part of the deliverable, not a nice-to-have.

## Template-Following Workflow

When creating a new deck from this template:

1. Inspect or render the canonical template before building.
2. Use its slide anatomy, colors, footer, page numbering, message bars, and
   card spacing as the visual contract.
3. Select slide patterns based on the teaching job, not on visual novelty.
4. Build the `.pptx`, render it, and inspect the contact sheet.
5. Fix misaligned boxes, clipped text, crowded cards, inconsistent message
   bars, missing trainer notes, and slides that do not feel like the template.
